# 浏览器从输入 URL 到页面渲染发生了什么？

## 推荐回答方式

这道题别一上来就散讲，最好按**时间顺序**拆成几个阶段：

1. URL 解析
2. DNS 解析
3. 建立连接（TCP / TLS）
4. 发送 HTTP 请求
5. 服务端返回响应
6. 浏览器解析并渲染页面
7. JS 执行、资源继续加载、页面更新

这样结构最清楚，也最像有工程经验的人在回答。

---

## 1）URL 解析

浏览器会先判断：

- 输入的是搜索词还是 URL
- 协议是否存在（`http` / `https`）
- 是否命中本地缓存 / HSTS / Service Worker 等策略

如果是一个标准 URL，浏览器会拆出：

- 协议
- 域名
- 端口
- 路径
- query
- hash

---

## 2）DNS 解析

目的是把域名解析成 IP 地址。

常见查找顺序可以概括为：

- 浏览器缓存
- 系统缓存
- 本地 hosts
- 路由 / ISP DNS
- 根域名服务器
- 顶级域服务器
- 权威 DNS

> 面试里不用把所有层级背成教科书，但要知道“DNS 的本质是域名到 IP 的映射”。

---

## 3）建立连接

### TCP 三次握手

在 HTTP/1.1、HTTP/2 的经典场景下，客户端和服务端通常先建立 TCP 连接。

核心目的是：

- 双方确认收发能力正常
- 协商初始序列号
- 建立可靠连接

### TLS 握手（如果是 HTTPS）

如果是 HTTPS，还会在 TCP 之上进行 TLS 握手，用于：

- 身份校验
- 密钥协商
- 后续加密传输

这里常被追问：

- HTTPS 比 HTTP 多了什么？
- TLS 握手为什么会增加耗时？
- HTTP/2 和 HTTP/3 在连接层面有什么差异？

---

## 4）发送 HTTP 请求

浏览器会组装请求报文，包括：

- 请求行（方法、路径、协议）
- 请求头
- Cookie
- 请求体（POST / PUT 等）

常见请求头示例：

```http
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Cookie: sid=abc123
```

---

## 5）服务端处理并返回响应

服务端收到请求后，一般会经历：

- 路由匹配
- 业务逻辑处理
- 访问数据库 / 缓存
- 拼装响应内容
- 返回状态码、响应头、响应体

比如：

- `200`：正常返回
- `301/302`：重定向
- `304`：命中协商缓存
- `404`：资源不存在
- `500`：服务端异常

---

## 6）浏览器解析页面并渲染

### HTML 解析 -> DOM Tree

浏览器会把 HTML 解析成 **DOM 树**。

### CSS 解析 -> CSSOM

样式会被解析成 **CSSOM**。

### 合成 Render Tree

DOM 和 CSSOM 会结合成 **渲染树**。

注意：

- `display: none` 的节点不会进入渲染树
- `visibility: hidden` 会进入渲染树，但不可见

### Layout（回流 / 重排）

浏览器计算每个节点的几何信息：

- 位置
- 尺寸
- 盒模型

### Paint（重绘）

把像素绘制到屏幕上。

### Composite（合成）

多个图层合成为最终页面。

---

## 7）JS 执行与资源阻塞

这部分是高频追问重点。

### 为什么 JS 会影响渲染？

因为浏览器在解析 HTML 时，如果遇到普通 `<script>`：

- 可能要先下载脚本
- 执行脚本
- 脚本可能修改 DOM / CSSOM

所以 HTML 解析可能被阻塞。

### CSS 为什么也会影响？

因为 JS 可能读取样式信息，浏览器通常会等 CSSOM 准备好后再执行相关脚本。

### `defer` 和 `async`

- `async`：下载完立即执行，执行顺序不保证
- `defer`：延迟到 HTML 解析完成后按顺序执行

---

## 可以顺手补充的性能点

如果你想答得更像“做过优化”的人，可以补：

- DNS 预解析：`dns-prefetch`
- 资源压缩：gzip / br
- 强缓存 / 协商缓存
- CDN 就近分发
- 图片懒加载
- 脚本拆包与按需加载
- `defer` / `async`
- 减少重排重绘
- SSR / 预渲染优化首屏体验

---

## 一段可直接背的面试回答

> 浏览器从输入 URL 到页面渲染，通常会经历：先解析 URL，然后做 DNS 解析拿到服务器 IP；如果是 HTTPS，还要先建立 TCP 连接并完成 TLS 握手；接着浏览器发起 HTTP 请求，服务端处理后返回 HTML、CSS、JS 等资源。浏览器拿到 HTML 后会解析出 DOM，拿到 CSS 后构建 CSSOM，再生成渲染树，随后进行 Layout、Paint 和 Composite，最终把页面显示出来。过程中如果遇到同步脚本，还可能阻塞 HTML 解析，所以实际性能优化里常会结合缓存、CDN、资源压缩、懒加载以及 defer/async 等手段优化首屏速度。
